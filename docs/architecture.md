# obs-nyan-real-3dof アーキテクチャ

本プロジェクトは、AR グラス（XREAL / RayNeo）の IMU データを使って通常のソースを
3DoF の仮想スクリーンに変換する、小さな OBS プラグインです。描画は仮想スクリーン
入力ソースが行います。

> このドキュメントは開発者向けの内部資料です（日本語が原則）。配布物には同梱しません。

## ソース構成

責務ごとに分割した複数の翻訳単位で構成します。共有型とレジストリ・トラッカーは
ヘッダー、デバイス I/O とUIは .cpp です。

```
src/
  plugin-main.cpp        モジュールエントリ（ロード/アンロードと各部の登録のみ）
  nyan_types.h           共有型（imu_transport / model_profile / 各サンプル / pose_snapshot）と定数
  math_util.h            クォータニオン・ベクトル数学とバイトリーダー（ヘッダーオンリー）
  head_tracker.h         姿勢トラッカー（on_imu / on_external_pose、ヘッダーオンリー）
  device_registry.h/.cpp 組み込みデバイステーブル + devices.json + EDID 識別子
  hid_io.h/.cpp          HID 列挙・機種判別と OVERLAPPED レポート I/O
  device_manager.h/.cpp  グローバル状態・ワーカースレッド・ドック向けセッター
  transports/            機種ファミリーごとの IMU セッション（transports.h に一覧）
    xreal_tcp.cpp / xreal_air.cpp(.h) / rayneo.cpp / moverio.cpp / rokid.cpp / viture.cpp / nreal_light.cpp
  cursor_fence.cpp(.h)   メガネ画面へのカーソル進入防止（WH_MOUSE_LL フック）
  dock.cpp(.h)           Qt ドック UI とプロジェクター制御
  virtual_source.cpp(.h) 仮想スクリーン入力ソースとワープシェーダー結線
  display-wall-source.cpp(.h)  Display Wall 入力ソース
  ws_server.cpp(.h)      自前 WebSocket サーバー（Audio Wall インジェストとリモコンで共用）
  remote_control.cpp(.h) スマホリモコン（LAN サーバー + SendInput 注入）
  remote_schema.cpp(.h)  リモコンにミラーするドック設定の定義表（追従ルールは CONTRIBUTING）
  qrcodegen/             ベンダリングした QR コード生成ライブラリ（Nayuki、MIT）
```

新しい機種ファミリーの追加は「`transports/` に 1 ファイル + `transports.h` に宣言 +
`device_registry.cpp` の組み込みテーブルに行を追加 + locale キー」が基本形です。
既存プロトコルの新 PID だけなら devices.json（後述）で足ります。

## ビルド依存

OBS Studio のソースはこのリポジトリにはベンダリングしていません。`setup.ps1` が
`deps/obs-studio` と `deps/obslib/obs.lib` を生成します。可能であれば
`..\obs_real3d\deps` を再利用します。ONNX Runtime、DirectML、モデルファイルは
意図的に使用していません。

## ランタイムのフロー

1. OBS が `obs-nyan-real-3dof.dll` をロードし、入力ソース ID
   `nyan_real_3dof_virtual_screen`、Display Wall ソース、`nyan Real 3DoF` ドック、
   recenter の frontend ホットキーを登録します。
2. モジュール単位の Device Manager がワーカースレッドを 1 本だけ起動します。
3. ワーカーはまず HID で対応デバイスを判別します。既知の HID デバイスが無い場合、
   TCP 接続も姿勢更新も開始せず、仮想スクリーンは参照元をそのまま表示します。
4. HID から機種が確定したら、装着オフセットと表示 FOV を決定します。
5. 現在の One-family 実装では、その後 XREAL One の TCP エンドポイント
   （既定 `169.254.2.1:52998`）へ接続します。ローカルに `169.254.2.*` の
   USB Ethernet アドレスがあれば、接続前にそのアドレスへソケットをバインドします。
6. ワーカーは生ストリームを、ヘッダー `28 36 00 00 00 80` を持つ 134 バイトの
   レコードとしてパースします。
7. `kind=11` のレコードはオフセット `34..57` にジャイロ/加速度を、`kind=4` の
   レコードはオフセット `58..73` に MAG/温度を提供します。
8. パースした IMU サンプルは mutex 下でグローバルなヘッドトラッカーを更新します。
   レンダースレッドは最新の姿勢スナップショットを読むだけで、ソケットには一切アクセス
   しません。複数の仮想スクリーンソースを配置しても HID/TCP 接続と姿勢推定は共有されます。
9. レンダースレッドは Device Manager から予測済みのクォータニオンと仮想スクリーンのパラメータを
   `data/nyan-real-3dof.effect` へ渡します。仮想スクリーンソースは参照先ソース/シーンを
   いったん `gs_texrender_t` へ描画してから、そのテクスチャへ同じワープシェーダーを適用します。
10. シェーダーはソーステクスチャを固定仮想スクリーンとして扱い、平板時は
    `z=-screen_distance_m` の平面、湾曲時は横方向の円筒面との交差によって
    逆方向サンプリングします。IMU が利用不可、未較正、または HID が切断された場合は、
    ソースをそのまま通過させます。

`Display FOV` の UI 値は対角の表示 FOV として解釈します。これは AR グラスの仕様の
慣例的な表記方法に合わせたものです。レンダラーはレイと平面の交差を行う前に、ソースの
アスペクト比から水平・垂直の投影タンジェントを導出します。

`Screen distance` と `Screen size factor` は、flinger の
`default_canvas_depth_meters` / `canvas_diagonal_fov_factor_array` に相当する概念です。
既定は `1.0m` / `0.25x`（`DEFAULT_SCREEN_DISTANCE_M` / `DEFAULT_SCREEN_SIZE_FACTOR`）。
距離の範囲は `0.1〜10m`（`MIN/MAX_SCREEN_DISTANCE_M`）。
1m 未満は「スクリーンに近づく」演出の実験域で、SBS 視差は距離に正しく追従するが、
メガネの焦点面は固定のため 0.5m を切ると輻輳調節矛盾（VAC）が強まる。
サイズ係数のスケール単位は固定の 4m（`SCREEN_SIZE_UNIT_DISTANCE_M`。係数 1.0 =
4m から見て FOV を満たす物理サイズ）で、距離スライダーとは独立です。この単位は
保存済み設定の見かけサイズを変えないため不変とします。50° 対角 FOV の場合、係数 1.0 の
基準対角サイズは `2 * 4.0 * tan(50° / 2) = 3.73m`、つまり約 147 インチ。既定の
`1.0m` / `0.25x` はその 1/4 の物理サイズ（約 37 インチ相当）を 1m 先に置くもので、
机上のモニターに近い初期状態になります（相似形のため mono の見え方は旧既定
`4.0m` / `1.0x` と同一）。サイズ係数はこの 4m 単位の物理サイズを線形に拡大縮小します。
距離を変えた場合は物理サイズを維持するため、見かけの角度サイズが変わります。これは
flinger の normal モードが `maintain_fov_on_depth_change: false` であることに
合わせています。機種プロファイルの `optics_focus_m`（光学虚像距離、未設定は 4m
フォールバック）は描画には使わず、SBS 時の快適距離としてドックの距離ツールチップに
案内を出すためだけの値です。

`Screen curve` は `0.0` で従来の平面、`1.0` で視点を中心にした円筒に近い横湾曲です。
上限は `3.0` で、かなり横長の参照元向けの強い湾曲として扱います。垂直方向は曲げず、
横長の参照元を見渡すときに左右端の読みやすさを上げるための設定です。

## UI/責務分担

- Dock: HID 検出状態、IMU 接続、Center/Recalibrate、FOV、スクリーン距離、
  スクリーンサイズ、湾曲、姿勢推定状態、デバッグ表示をグローバルに管理します。
  UI は折りたたみ式セクション(ステータス / デバイス操作 / メガネ画面・出力 /
  仮想スクリーン / スマホリモコン / 詳細設定。ヘッダーは矢印左端・タイトル中央の
  クリック可能バンド)に分かれ、開閉状態は保存対象(`dock_collapsed` ビットマスク、
  既定はスマホリモコンと詳細設定を折りたたみ)です。スマホリモコンセクションは
  有効化トグル・ポート・QR コード/URL 表示で、サーバー本体は remote_control.cpp
  (専用の節を参照)が持ち、ドックのポーリングが `remote_control_sync()` を駆動します。ステータスを畳むとヘッダーに「● 機種名」
  (緑=較正済みで追従中・黄=接続/較正中・赤=未検出か追従 OFF。状態名はツールチップ)を
  要約表示します。姿勢追従の ON/OFF は最上部の目玉トグル(OBS テーマの
  indicator-visibility クラスでソース一覧の目玉と同じ見た目)で、常時表示です。
  機種の手動選択 UI はありません。機種は常に HID 自動判別で、判別の誤り/未対応機種は
  ユーザーの `devices.json`(後述)で補正・追加します。
  transport 固有の設定行(現状は One-family の IP/ポート。「詳細設定」内)は
  `transport_traits` テーブルを参照して表示/非表示を切り替えます。未検出時は
  transport 固有行をすべて隠し、機種固有の操作行が 1 つもない機種では
  「デバイス操作」セクションごと非表示にします。
  ステータスの「IMU 経路」行は検出機種の transport(`detected_transport_for`)を表示します。
  「メガネへ全画面表示」はメガネ画面が未検出の間は無効化(グレー)し、非表示には
  しません(HID とメガネ画面は独立した検出軸のため、機能の存在とステータス行による
  理由説明を残す)。
  「検出時に自動で全画面表示」は次の接続への事前設定なので常時表示・操作可能です。
  さらに EDID で判別したメガネのディスプレイ(後述)を監視し、「メガネ画面」ステータス行、
  「メガネへ全画面表示」ボタン、「検出時に自動で全画面表示」(接続ごとに 1 回のラッチ)を
  提供します。プロジェクターのモニター番号は Qt 6.4+ の `QScreen::name()` 変更に対応した
  3 段階マッチング(GDI 名 → EDID フレンドリー名 → ネイティブ座標)で解決します。メガネ
  画面上の既存プロジェクター(手動で開いたものを含む)は置き換えて多重に開かず、メガネの
  ディスプレイが切断された場合はそこにあったプロジェクターを自動で閉じます(Windows が
  ウィンドウを他のモニターへ移動させ、デスクトップを覆うのを防ぐため)。監視はメガネ画面の
  実解像度もキャッシュし、仮想スクリーンのレンダー解像度の自動追従に使われます。
  「音声の出力先」コンボ(既定「メガネに自動」)は OBS のモニタリングデバイスの
  扱いを選ぶ(PC で聞く音の出力先であり、配信・録画ミックスには影響しない)。「メガネに自動」は機種検出中にブランド名トークンでメガネの USB
  オーディオ端点を探して切り替える(接続ごとに 1 回のラッチ)。特定デバイス選択時は
  モニタリングをそのデバイスへ向け続ける — WASAPI 端点 ID(再接続でも不変)+表示名で
  保存され、端点が不在の間(Bluetooth イヤホンの電源 OFF など)も選択は保持されて
  コンボに「(未接続)」サフィックス付きで表示され、再列挙され次第ラッチ再武装で
  自動復帰する(不在中は OBS のモニタリング設定に触れない)。
  「切り替えない」はプラグインがモニタリングを管理しない。
  いずれの適用パスも、既に同一デバイスが設定済みの場合に
  `obs_reset_audio_monitoring()` を 1 回呼ぶ — OBS 起動とデバイスの列挙が
  競合すると、モニター初期化が AUDCLNT_E_DEVICE_INVALIDATED (88890004) で失敗し
  **OBS は自前でリトライしない**ため、「端点の存在を確認できた瞬間」に全モニターを
  再構築して救済する(実機で遭遇した起動競合への対策)。
  「メガネ画面にカーソルを入れない」(既定 OFF)は WH_MOUSE_LL フックによる
  カーソルフェンスです。カーソルがメガネ画面に入ろうとしたとき、貫通した先に
  通常ディスプレイがあればそこへワープ(トンネル。メガネ画面が通常画面の間に
  挟まる配置でもカーソルの連結性が保たれる)、無ければ壁としてブロックします。
  ワープは SendInput の dwExtraInfo 署名でフック再入を素通しします。フェンスは
  メガネ画面の EDID 検出中だけ立ち、有効化時にカーソルがメガネ画面内にいた場合は
  通常ディスプレイへ押し出します。押し出し先が無い構成(通常ディスプレイなし)では
  カーソル詰み防止のためフェンスを立てません。フックはドックのポーリングが
  管理するため、Qt なしビルドでは機能しません。
- Device Manager: Dock の設定を保存/復元し、HID/TCP/トラッカー/姿勢スナップショットを
  モジュール単位で保持します。
- Virtual Screen Source: `OBS_SOURCE_TYPE_INPUT` / `OBS_SOURCE_CUSTOM_DRAW` /
  `OBS_SOURCE_COMPOSITE` として登録します。ソース固有の設定は、参照する OBS ソース/シーン名
  だけです。OBS へ出力するビューのレンダー解像度は自動で、メガネ画面の接続中は EDID 判別した
  ディスプレイの実解像度(ドックの監視がキャッシュし、`video_tick` は atomic 読みのみ)、
  未接続時は HID 判別デバイスのプロファイル解像度に追従します。プロパティには現在値を
  情報表示として出すだけです。参照元の `gs_texrender_t` へのキャプチャはフレーム内で
  キャッシュされ、プレビュー/プログラム/プロジェクターの複数ビューから描画されても
  ウォール全体の再レンダリングは 1 フレーム 1 回です。参照元を OBS の子ソースとして列挙し、
  表示中に参照先が変わる場合は
  `obs_source_add_active_child` / `obs_source_remove_active_child` を使います。自分自身を含む
  シーンや別の仮想スクリーンを参照しないようにして、再帰描画と二重ワープを避けます。
- Display Wall Source: `src/display-wall-source.cpp` に分離した入力ソースです。Windows の
  アクティブディスプレイを列挙し、OBS 標準の `monitor_capture` を private child source として
  複数作成します。列数による自動配置、行レイアウト文字列、Windows のディスプレイ座標、隙間、
  余白、短い行の寄せをもとに合成後サイズを返し、描画時に各 child を計算済み座標へ配置します。
  Windows 配置モードでは選択ディスプレイの左端/上端を列/行として扱い、後続列/行に設定隙間を
  加算して OS 配置を保ったまま余白を作ります。
  検出一覧の番号は `EnumDisplayMonitors` の列挙順ではなく、`QueryDisplayConfig` の active path
  順から得た Windows 識別番号を使います。`OBS 出力サイズを Display Wall に合わせる` が有効な
  場合は、合成サイズの安定後にプロファイルの `Video/BaseCX` / `BaseCY` / `OutputCX` / `OutputCY`
  を更新して `obs_frontend_reset_video()` を呼びます。
  既定値は実用構成(Windows の配置、プライマリディスプレイを含む、縦横の隙間 1 px、
  カーソルキャプチャ ON)です。EDID から対応 AR グラスと判別したディスプレイは
  「メガネのディスプレイを自動除外」(既定 ON)で wall から外し、ソースプロジェクター経由の
  合わせ鏡を防ぎます。追加の除外は従来どおり除外フィルターで行います(OBS の公開 frontend
  API では既存ソースプロジェクターの出力先を列挙できないため)。ディスプレイ構成の監視は
  毎秒の完全列挙ではなく安価なトポロジ署名の比較で行い、変化したときだけ
  `QueryDisplayConfig` などの再列挙を実行します(映像スレッドの周期的なヒッチ防止)。
  VDD の作成/削除や入力注入はまだ責務に含めず、将来的に別プロセス/別プロジェクトへ
  分離しやすい境界にしています。

## トラッカー

トラッカーは `xrealonenet/3dof/js/app.js` の `headTracker` のフローをミラーします。

- 静止サンプルからの初期ジャイロバイアス較正。ストリーム開始後 0.5 秒のウォームアップを
  置き、平均窓（12 サンプル以上かつ 2 秒）内で動きを検出した場合（ジャイロ標準偏差
  > 0.03 rad/s または加速度ノルム相対標準偏差 > 0.05）は窓を破棄してやり直します。
  これは参照実装に対する意図的な拡張です。動いたまま推定したバイアスはドリフトを生むだけで
  なく、静止判定（バイアス補正後のジャイロと閾値 0.05 rad/s の比較）を恒久的に外し、
  バイアス自動適応までロックアウトするため、悪い窓を確定させないことが重要です。
  やり直し中はドックの姿勢が「較正中」のまま維持されます。
- 装着オフセット（X 軸回りの自由角度）。既知値: XREAL One Standard +180 度、
  XREAL One Pro -150 度、XREAL Air 0 度、RayNeo Air -20 度、MOVERIO BT-40 / BT-30C 0 度、
  Rokid Max -4 度 / Rokid Air -1.3 度（参照実装のチルト 0.07 / 0.022 rad より）、
  VITURE One +6 度 / VITURE Pro +3 度（XRLinuxDriver のピッチ補正より）、
  Nreal Light +15.2 度（ar-drivers-rs のチルト -0.265 rad より。符号は Rokid 実機で
  確認した変換規約に従う）
- ジャイロ積分
- 加速度計による重力補正
- 静止時のジャイロバイアス自動適応
- 逆姿勢による正面リセット（recenter）
- 任意の緩やかな MAG ヨードリフト補正
- 描画時の姿勢予測

MAG 補正は既定で無効です。局所的な磁気干渉がヨードリフトより悪化し得るためです。
これは絶対的な北固定モードではなく、緩やかな相対ヨー補正です。

頭部並進（6DoF）は扱いません。マーカー（AprilTag）による 6DoF 拡張を
v0.6 系で実験しましたが、タグの印刷・設置が必要で XREAL Eye 専用という
セットアップの重さに対して体験が見合わず、v0.6.2 で削除しました（経緯と
精度検証は MyGlasses2.0/analysis/18 を参照。「近づいて大きく見る」の
代替はスマホリモコンの視線ドリー — コマンド駆動の視点並進 — として
提供済み）。IMU からの実頭部並進の推定は引き続き行いません。

### 外部姿勢入力（本体フュージョン機種）

VITURE のように本体でフュージョン済みの姿勢を送ってくる機種は、ジャイロ積分・
較正・バイアス適応をすべてバイパスする `on_external_pose` 経路を使います。受け取った
クォータニオンに装着オフセットを右掛け（フレーム共役）して `q_` に直接格納するため、
接続直後から常に「較正済み」です。描画時予測に使う角速度は連続する姿勢の差分
（小角近似）から推定します。リセンタリングは通常機種と同じ逆姿勢方式です。

## デバイス判別 (HID)

機種は Windows の HID API（`SetupAPI` + `hid.dll`）で USB の VID/PID と UsagePage を列挙して
判別します。HID はランタイムの前提条件です。機種の手動選択 UI はありません。

判別はデバイスレジストリ（`g_device_registry`）への先勝ちマッチで行います。レジストリは
`obs_module_load` で一度だけ構築され（ユーザーエントリ → 組み込みテーブルの順）、以後は
不変なので全スレッドがロックなしで読みます。組み込みの VID/PID は
`xrealonenet/3dof/js/app.js` のデバイス表をミラーしたものです。

各エントリは `{vid, pid, product_contains, profile}` で、`pid = 0` は同一 VID の全 PID に
マッチするワイルドカード、`product_contains` は HID ProductString の部分一致（大文字小文字
無視）による絞り込みです。profile は transport / 装着オフセット（X 軸回りの度数）/ FOV /
表示解像度を持ちます。devices.json の `mount` はプリセット名
（`standard` / `pro` / `air` / `rayneo`）でも度数の数値でも指定できます。

### ユーザー拡張 (devices.json)

`<OBS設定>/plugin_config/obs-nyan-real-3dof/devices.json` を置くと、再ビルドなしで
機種を追加・上書きできます。書式は `data/devices-example.json` を参照してください。
ユーザーエントリは組み込みより先にマッチするため、組み込みプロファイルの上書きにも
使えます。エントリ単位でバリデーションし、壊れたエントリは警告ログを出してスキップ
します（組み込みテーブルは常に残ります）。既存プロトコルの新 PID（例: Air 系の新機種）は
エントリをコピーして PID を書き換えるだけで対応できます。新しいプロトコルが必要な機種は
transport の追加（コード対応）が必要です。エントリには表示パネルの EDID 識別子
（`edid_vendor` / `edid_product` / `edid_name_contains`）も書け、次節のメガネディスプレイ
判別に使われます。

### メガネディスプレイ判別 (EDID)

IMU の機種判別（USB HID）とは別に、メガネの表示パネルを EDID（製造者 PNP ID +
プロダクトコード + モニター名）で判別する共通基盤があります。組み込みの識別子は
XREAL = ベンダー `MRG`（全世代一致）、RayNeo Air = `TCL` 03D4 / モニター名
"SmartGlasses"、EPSON MOVERIO = `SEC` D004 (BT-40) / D003 (BT-30C)（SEC は
エプソンのプロジェクターとも共有のためプロダクトコードで限定。いずれも実機確認）、
Rokid = `LBT` + モニター名 "Rokid"、
VITURE = `CVT` + モニター名 "VITURE"（CVT は汎用ベンダー ID のためモニター名で限定）、
Nreal Light = ベンダー `NRL`（MRG 以前の Nreal 専用 ID。実機確認: NRL 0x3132 /
モニター名 "nreal light"）です。
この判別は次の 2 箇所から参照されます。

- Display Wall の「メガネのディスプレイを自動除外」
- Dock の「メガネ画面」ステータス、「メガネへ全画面表示」/自動全画面、切断時の
  プロジェクター自動クローズ、仮想スクリーンのレンダー解像度の自動追従

devices.json の `edid_*` フィールドで判別対象を拡張できます。

IMU 受信 transport は機種プロファイルで分岐します。

- One-family (`one_bridge_tcp`): USB-Ethernet の TCP ブリッジ (`169.254.2.1:52998`) から
  134 バイトレコードを受け取る。有効レコードが 3 秒途絶えた場合は切断して自動再接続する
  （Air/RayNeo のセッション内リトライに相当）。セッションは XREAL Eye カメラ用の
  コントロール HID も開く（1024 バイトレポート、Air MI_04 と同じ 0xFD フレーミング —
  `build_xreal_control_packet` / `xreal_control_exchange` として共有化。インターフェース
  特定はカメラステータス照会 0x00D5 への応答で行う）。Eye の装着状態（応答データ
  先頭バイト 0x00 = 装着）と UVC0/UVC1 状態（0x00D2、u32 の bit12-13/14-15）を
  5 秒間隔でポーリングして `eye_present` / `eye_uvc` に公開し、ドックの「Eye カメラ」
  行とトグルボタン（`eye_request` 経由、0x00D3 SET）を駆動する。SET 後はメガネが
  USB を再列挙して TCP ごと切断されるため、再接続パスが状態を再取得する
  （プロトコル事実は denkimeganetool/eyecon の WebHID 実装より。実機検証待ち）
- Air-family (`air_hid`): Windows HID へ START IMU コマンドを送り、入力レポートから
  IMU/MAG を直接受け取る。表示モード切替用に MI_04 のコントロール HID も開く
  （後述「表示モード切替と SBS 出力」）。ストリーム IF の特定はヘッダ判定ではなく
  実デコード成功で行う — Air 2 には常時カウンター列を流す書込不可の MI_05 があり
  （列挙順は MI_03 より先）、先頭バイトが 256 回に 1 回 0xAA を通過するため、
  ヘッダ判定だとこの IF を掴んで無サンプルのまま「較正中」で止まることがある
  （Air 2 実機で確認・修正済み）
- RayNeo Air-family (`rayneo_hid`): Windows HID へ `[0x66, cmd, arg]` 形式のコマンド
  （ACQUIRE_DEVICE_INFO / OPEN_IMU / CLOSE_IMU）を送り、入力レポートのバイト列から
  `[0x99, kind, len, ...]` フレームを再組み立てして float32 の IMU/MAG を受け取る。
  `xrealonenet/3dof/js/rayneo_driver.js` のミラー。XREAL Air と異なり参照ドライバーは
  軸の入れ替えをしないため、デコーダーも軸変換なしでトラッカーへ渡す
- EPSON MOVERIO BT-40 / BT-30C (`sensor_api`): IMU が標準の HID センサーコレクションとして
  列挙されるため、Windows Sensor API（`ISensorManager`）でジャイロ（deg/s）・
  加速度（G）・地磁気（mG）をポーリング取得する。検出した USB 識別子をデバイスパスに
  含むセンサーだけを選ぶため、他デバイスのセンサーを掴むことはない。レポート間隔は
  最小値を要求し、レポートのタイムスタンプ変化でサンプルの鮮度を判定する。
  MOVERIO の公式ガイドによると加速度センサーだけ軸が反転している
  （ジャイロ/地磁気: X=右・Y=上・Z=装着者方向、加速度: X=左・Y=下・Z=視線方向）
  ため、加速度を全軸符号反転して共通フレームに揃える。
  3 秒間サンプルが来なければ切断して自動再接続する。
  デバイスの COM ポート（USB シリアル、MI_01）は SDK の Windows 用コマンド
  リファレンスにあるディスプレイ制御コマンド専用で、IMU には使わない。セッションが
  このポートも開き、`getbright` で現在輝度を取得、Dock の「画面の明るさ」
  （0〜20、transport traits の `display_brightness` で MOVERIO 接続時のみ表示）からの
  変更要求を `setbright` で適用して `getbright` で検証する（応答のフレーミングが
  未文書化のため、応答文字列は信用せず再取得で確認する方式）。「自動調光」
  チェックボックスは `enableautobright 0/1` / `getautobright` に対応し、環境光
  センサーによる自動輝度調整を切り替える（ON の間、手動スライダーは無効。自動調光中の
  `getbright` は 50 を返すためスライダーへの同期も自然に止まる）。ポートが開けない場合、
  IMU はそのまま動き、明るさ・自動調光の行はグレーアウトする。
  「輻輳をスクリーン距離に連動」チェックボックスは `setdisplaydistance`（左右画像の
  水平シフト、-32〜256px。0 = 工場既定で輻輳 4.6m、正で近づく）で映像面の見かけの
  奥行きをドックのスクリーン距離に合わせる。SDK 付録のステップ表は
  px = 235 × (1/距離 − 1/4.6) の幾何式に一致するためこの式で連続変換し、
  スライダー操作中は 200ms 間隔のレート制限付きで追従、OFF に戻すと既定 0 を再送する。
  本コマンドは BT-40 のみ対応（BT-30C 非対応。機種プロファイルの `display_distance`
  でゲートし、対応機のみ接続時に `getdisplaydistance` で現在値を取得。取得に失敗した
  場合もグレーアウト）
- Rokid Max / Air (`rokid_hid`): ベンダー定義 HID（MI_02）の 64 バイト固定パケットを
  読む。開始コマンド不要の常時ストリーム。type 4 = 単一センサー（byte1: 1=加速度
  2=ジャイロ 3=磁気、µs の u64 タイムスタンプ@9、f32×3@21。加速度をキャッシュし
  ジャイロ到着時にペアで発行）、type 17 = 結合（ns の u64 タイムスタンプ@1、
  加速度@9・ジャイロ@21・磁気@33）。値は SI 単位（rad/s、m/s²）の共通フレームで
  軸変換も deg→rad 変換も不要（プロトコル事実は ar-drivers-rs の Rokid ドライバーを
  参照。コードの移植はしていない）。再開コマンドが無いため 3 秒無サンプルで切断して
  自動再接続する。実機は約 440 Hz
- VITURE One / Pro 系 (`viture_hid`): ベンダー定義 HID が 2 つあり（MI_00 = IMU、
  MI_01 = MCU コマンド）、生 IMU ではなく本体フュージョン済みのオイラー角
  （度、ビッグエンディアン f32×3 @18、順序 roll/pitch/yaw）を 64 バイトパケット
  （ヘッダー FF FC）で受け取る。コマンド 0x15（データ 1 バイト: 1=開始 0=停止）を
  全インターフェイスへ送って開始し、FF FC が流れてきたインターフェイスを IMU 側と
  特定する。パケットの CRC-16-CCITT（多項式 0x1021、初期値 0xFFFF、オフセット 4 以降）
  はバイト順が参照実装間で食い違うため、コマンドは両順序で送る（不正 CRC は無視され、
  開始コマンドは冪等）。受信姿勢はトラッカーの外部姿勢入力（前述）へ渡す。
  1.2 秒無受信で開始コマンドを再送、3 秒で切断して自動再接続する。実機は 60 Hz
  （プロトコル事実は bfvogel/viture-webxr-extension の文書と
  mgschwan/viture_virtual_display を参照。コードの移植はしていない）
- Nreal Light (`nreal_hid`): Nreal Light は複数の独立 USB デバイスの集合体で、
  6DoF IMU は MCU ではなく OV580 カメラコプロセッサー（05A9:0680、レポート ID 付き
  ベンダー HID）にある。コマンドは 7 バイトレポート `[レポート ID 0x02, cmd,
  subcmd, 0...]`（レポート ID を前置する共有ヘルパー `hid_write_report` は使えない
  ため専用の書き込みを持つ）。cmd 0x19 subcmd 1/0 で IMU ストリームを開始/停止。
  ストリームはレポート ID 0x01 の入力レポートで、オフセット 44 から u64 LE ジャイロ
  タイムスタンプ（ns）・u32 倍率・u32 除数・i32×3（mul/div 度/s 単位）、続いて同形式の
  加速度ブロック（mul/div g 単位）。ジャイロは deg→rad、加速度は ×9.81 で SI に変換し、
  軸は (x, -y, -z) で共通フレームへ写像（ar-drivers-rs の nreal_light ドライバーで
  実証済みの写像。コードの移植はしていない。2026-06 実機確認）。OV580 コンフィグ内の
  工場バイアスは読まず、トラッカーの静止較正に任せる。1.2 秒無受信で開始コマンドを
  再送、3 秒で切断して自動再接続する。メガネは未装着・映像なしが続くとスリープして
  USB ハブごと切断されるため、ストール検出が再接続の入り口になる。実機は 1000 Hz。
  磁気センサーは別デバイス（STM32 MCU の HID、0486:573C）のテキストプロトコル側に
  あるため未対応（MAG ヨー補正は利用不可）。セッションは表示モード切替用に MCU の
  HID も開く（後述）

### 表示モード切替と SBS 出力

機種ファミリーが対応する表示モード（2D/3D SBS、リフレッシュレート）は
`transport_traits.display_modes` に列挙され、ドックの「表示モード」コンボは
このリストから生成される（空なら行ごと非表示 — 機種別 UI の汎用機構）。選択は
`display_mode_request` に積まれ、セッションがコマンドチャネルで適用し、GET で
`display_mode_current` を更新する（-1 = チャネルなし、行はグレー）。

- Air-family: コントロール HID（MI_04、非番号レポート）。フレーミングは
  `[0xFD][CRC32 LE][payloadLen u16][timestamp 8B][msgId u16][?5B][data]`、CRC は
  オフセット 5 から payloadLen バイト。msgId 7 = GET / 8 = SET（データ 1 バイト =
  モード値）。応答は status@22・データ@23。モード値: 1=2D60 / 5=2D72 / 10=2D90 /
  11=2D120 / 3=SBS60 / 4=SBS72（Air 2 / Air 2 Pro 実機確認、
  MyGlasses2.0/analysis/12 参照。コードの移植はしていない）。72/90Hz 系は OLED の
  デューティ比も下がり低残像になる。Half-SBS(8) と SBS90(9) は SET は通るが
  画面が表示されないためリストから除外（8 = Air 2 Pro、9 = Air 2 実機確認）
- Nreal Light: STM32 MCU の ASCII プロトコル
  （`\x02:type:code:data:TS:CRC32:\x03`、64 バイト固定、CRC は TS 後のコロンまで、
  TS はリテラル "0" 可）。コード '3'、GET type '3' →「`1&2D_1080`」等、SET type '1' +
  ASCII 数字。モード 1=2D60 / 3=SBS60 / 4=SBS72（4 は ELLA リビジョンのみ。540p の
  モード 2 は除外）。SBS 3D モードには SDK ハンドシェイク（`@3:1`）が必要で、動作中は
  ハートビート（`@K`、250ms 間隔）を切らすと画面が消えるため、セッションが接続中
  常時送信する。応答は category が type+1 で返る（ar-drivers-rs の実装と実機検証
  済みの解析メモに基づく）

- VITURE: IMU と同じ 2 つのベンダー HID（FF FE コマンド / FF FD 応答、CRC-16-CCITT）。
  GET = cmd 0x07（データなし）、SET = cmd 0x08 でデータ 1 バイト '1'(0x31)=2D 1920x1080 /
  '2'(0x32)=3D SBS 3840x1080。応答ペイロード先頭バイトがその ASCII 状態を返す
  （公式 Linux SDK 1.0.7 の set_3d / get_3d_state を逆アセンブルして確認。プロトコル
  事実のみ利用、SDK コードは不使用）。VITURE にはリフレッシュレート切替コマンドは無く、
  この解像度トグルのみ。`display_modes` の値は ASCII 状態バイトそのもので、セッションは
  接続時と SET 後に GET して `display_mode_current` を更新する。3D に切替えると画面が
  3840x1080 になり SBS 出力「自動」が追従する

Rokid にも切替コマンド自体は存在する（ar-drivers-rs: EP0 ベンダー要求 GET 0x81 /
SET 0x01、wIndex 1、モード 0=2D / 1=SBS / 3=2D 高リフレッシュ / 4=SBS 高リフレッシュ）
が、Windows では全インターフェースが標準クラスドライバ（usbaudio / HidUsb）に
バインドされており WinUSB のベンダー IF が無いため、ドライバ差し替えなしには
送れない（Rokid Air 実機で構成確認、2026-06-11）。よって `rokid_hid` の
display_modes は空のまま。本体ボタンでの 2D/3D 切替は EDID 再列挙で SBS 出力の
「自動」が追従する。

SBS 出力（描画側）はデバイス非依存で、`sbs_output`（0=自動 / 1=オン / 2=オフ、
ドックの「SBS出力」コンボ、保存対象）に従い仮想スクリーンの video_render が
ワープ済みビューを左右半分に 1 回ずつ描く。「自動」は
EDID 判定済みメガネ画面の実モードが縦の 3 倍以上の横幅（3840x1080 = 3.56 等）に
なったとき有効。Half-SBS は解像度から検出できないため手動オン専用。XREAL One の
ように本体操作で SBS にした場合も実モード追従で自動有効になる。
per-eye の FOV/アスペクト計算に使う幅は SBS の種別で変える: フル SBS
（横倍幅フレーム）は各半分がパネルへ 1:1 で写るため半分幅、Half-SBS
（通常アスペクトのフレーム、横幅 < 縦×3 で判定）はメガネが各半分を
パネル全幅へ引き伸ばすため**フレーム全幅**で計算し、横 1/2 に圧縮した
アナモルフィック描画にする（半分幅で計算すると引き伸ばし後に横太りに
見える）。

SBS の左右は同一画像ではなく、眼ごとの視差を付けて描く。シェーダーのレイ原点
`eye_pos_m` に、頭部フレームの ±IPD/2 横オフセットを姿勢クォータニオンで
ワールドへ回したものを渡し（左半分 = 左眼 = -X 側）、ドローの間で
`gs_effect_set_vec3` を呼び替える（libobs は device_draw ごとに
`gs_effect_update_params` を呼ぶため、パス内のパラメータ差し替えが効く）。
これで仮想スクリーンは光学的無限遠ではなく `screen_distance_m` の距離に
立体的に定位し、距離が近いほど両眼が寄る。IPD はドックの「瞳孔間距離 IPD」
（既定 63mm、40〜80mm、保存対象）。IPD 行は SBS 出力が実際に有効なときだけ
操作可能（無効時はグレー。判定はレンダラーの `sbs_output_active` を共有し、
メガネ画面の実モード/手動オンの両方に追従する）。SBS 無効時は視点オフセットのみの
単眼描画。平面・円筒の両交差ともレイ原点対応で、リモコンの視線ドリー
（`viewer_offset_x/y/z`、「スマホリモコン」の節を参照)がこの `eye_pos_m` への
加算として「近づくと大きく見える」を実装している。将来 6DoF の実頭部並進も
同じ加算で済む。

判別は専用の検出スレッド（`detect_worker_fn`）が約 1 秒ごとに実行し、結果を
`detected_model`（レジストリの 1 始まりインデックス、0 = unknown）に保持します。
全 HID インターフェースの列挙（SetupDi ＋ デバイスごとの open・文字列・caps 取得）は
数十 ms かかり、IMU 読み取りスレッドで同期実行すると 1000 Hz 機種（Nreal Light）では
既定 32 レポートの HID 入力キュー（= 32 ms 分）を使い切って毎秒 1 回の姿勢ヒッチに
なるため、セッションループからは分離しています（セッションは atomic を読むだけ）。
あわせて `open_hid_path_rw` は `HidD_SetNumInputBuffers` でキューを 128 レポートへ
拡大し、OS スケジューリング起因の取りこぼしにも余裕を持たせています。接続中に HID が
unknown へ戻った場合、または機種に対応する transport が変わった場合は、現在の接続を
閉じ、姿勢スナップショットを無効化します。

## 空間音響フィルター

`src/spatial_audio_filter.cpp` の音声フィルター（`nyan_real_3dof_spatial_audio`）は、
音源を「リセンター後の正面を 0° とした仮想空間の方位」に固定し、共有ポーズの
ヨーに対する相対方位で定電力パンニングする。映像のキャプチャ自体は無音なので、
アプリケーション音声キャプチャ等に掛けて使う（画面と音声の対応付けはユーザーが
方位角スライダーで行う。±90°、右が正）。

- ヨーは `yaw_from_quat_heading`（左が正）を音声スレッドから state_mutex の
  短いロックで取得。ポーズ無効時は正面扱いで設定方位に置いたままにする
- 定位モードは 2 つ: 点音源（L+R をモノ化してパン。スクリーン=1 音源の想定で
  定位が明確）/ バランス（ステレオ維持、遠い側だけ減衰。中央で利得 1）
- パン引数は全周の sin(相対方位) で円周上連続: 背面は前面の鏡像として中央へ
  折り返す（実際の両耳レベル差と同じ対称性）。±180° で逆の耳へジャンプする
  不連続が無い（クランプ式 sin(clamp(±90°)) からの修正、実機の「真後ろで音が
  ワープする」報告による）
- 背面（|相対方位| > 90°）は最大 -2.5dB 落とす簡易キュー。パンが鏡像のため、
  前後の区別はこのレベル差が担う
- 「スクリーン距離で音量を変える」はドックの距離への逆距離則（既定距離
  `DEFAULT_SCREEN_DISTANCE_M` 基準でニュートラル、-12〜+6dB にクランプ）。
  将来の 6DoF で「近づくと大きく聞こえる」に直結する
- ゲインはブロック内線形ランプでジッパーノイズを防止。モノラル入力はパン不能
  なので距離/背面減衰のみ、3ch 以降も距離/背面のみ適用
- セットアップ自動化: `filter_add` で親ソースのモニタリングが NONE なら
  「モニターのみ」へ自動昇格（明示的なユーザー設定は上書きしない）。ドックの
  「音声の出力先」（既定「メガネに自動」、保存対象）は、機種検出中に
  ブランド名トークン（xreal/nreal/viture/rokid/rayneo/moverio。EPSON は
  プロジェクターと衝突するため不使用）でメガネの USB オーディオ端点を探し、
  OBS のモニタリングデバイスを切り替える（接続ごとに 1 回のラッチ。USB
  オーディオは HID より遅れて列挙されることがあるため、見つかるまでポーリング
  ごとに再試行）。イヤホンなど特定デバイスの選択・不在時の保持・自動復帰は
  上記「音声の出力先」コンボの節を参照。残る手動手順は
  「Windows の既定出力をメガネ以外にする」のみ
  （アプリ本体の直接再生は OBS を通らないため、ソフト側では消せない。
  セッションミュートはプロセスループバックキャプチャ自体を無音化するため不可）
- v2 候補: ITD（±0.6ms の左右遅延）、背面ローパス。v3: HRTF 畳み込み
  （上下定位と外在化）
- パン計算（ヨー取得・定電力パン・背面/距離減衰）は `spatial_pan.h` に
  ヘッダーオンリーで共有され、フィルターと Audio Wall の挙動が一致する

## Audio Wall（Display Wall の空間音響グループ）

`src/audio-wall-source.cpp` は Display Wall の音声版で、「内部プライベート
子ソース」の同じパターンを音声に適用したもの。かつては独立ソース
（`nyan_real_3dof_audio_wall`）だったが、方位の根拠（ウォールレイアウト）と
表示が同じものを 2 ソースに分ける意味がないため、Display Wall ソースの
チェック可能プロパティグループ「空間音響 (Audio Wall)」（設定キー
`audio_wall`、既定 OFF）として統合した。実装はエンジン API
（`audio_wall_engine`、create/destroy/update/tick/show/hide/enum）で、
Display Wall が自分のソースコールバックから委譲する。ユーザーは Display
Wall のプロパティでチェックを 1 つ入れるだけでよい。

- ポーリングスレッド（2 秒間隔、COM MTA）が WASAPI セッション列挙
  （`IAudioSessionManager2`、全アクティブ出力デバイス横断、システム音と
  obs 自身は除外）で「音を再生中の exe」を発見し、EnumWindows でその exe の
  最大ウィンドウを特定（音声セッションはブラウザ等で子プロセスに属するため
  exe 名でマッチ）。ウィンドウ中心の仮想デスクトップ上の正規化 X（-0.5..0.5）を
  方位の素とする。最小化中は `GetWindowPlacement` の復元位置を使う
- UWP/ストアアプリ（メディアプレイヤー等）はトップレベル窓を
  ApplicationFrameHost.exe が所有し、アプリ本体の exe は子ウィンドウしか
  持たないため、exe 直接マッチに失敗したら ApplicationFrameHost のフレーム窓の
  子ウィンドウ PID を辿ってフォールバックする（位置はフレーム窓の矩形、
  window 文字列の exe はアプリ本体 — win-wasapi の UWP 対応マッチングが解決する）。
  発見結果は変化時に `audio wall: capturing: ... / no window: ...` として
  ログされ、スキップが黙って起きない
- 結果は desired リスト + 世代カウンタに置かれ、video_tick が差分適用する:
  新 exe は `wasapi_process_output_capture` のプライベート子ソースを生成
  （window 文字列は win-wasapi の `#3A` エスケープ、priority=EXE でタイトル
  変化に強い）、消えた exe は解放。active child 管理は Display Wall と同じ
  show/hide + ラッチ方式
- **ミックスは自前で行わない**。当初は OBS_SOURCE_COMPOSITE で scene_audio_render
  方式のミックスを実装したが、(1) libobs は COMPOSITE|AUDIO の登録を拒否する、
  (2) コンポジットの音はモニタリング経路に乗らない（シーンをモニタリング
  できないのと同じ）ため、メガネで聴く本用途では成立しない。代わりに各子
  キャプチャへ空間音響フィルター（`nyan_real_3dof_spatial_audio`）をプライベート
  生成して `obs_source_filter_add` し、フィルターの filter_add が子を
  「モニターのみ」へ昇格させる。各子が個別にモニタリングへ出て、モニタリング
  デバイス上で自然に合成される。エンジン自体は AUDIO/VIDEO に関与しない管理層
  （ホストの Display Wall も空間音響の音声は出力しない）
- 方位の決定は既定で「映像から自動計算」: 仮想スクリーンのレンダーパスが
  毎フレーム公開する物理半幅（`g_device.screen_half_width_m`）+ 距離 + 湾曲と、
  Display Wall がレイアウトごとに公開する「デスクトップ矩形 → テクスチャ u」
  対応表（`nyan_real_get_wall_monitor_map` / 世代カウンタ）から、
  窓位置 → ウォール u → スクリーン上のワールド座標 → `atan2` で方位角を厳密に
  解く（湾曲時はシェーダーと同じ円筒を使い、arc 長→θ→方位）。距離・サイズ
  係数・湾曲・FOV・ウォール構成・センター回転の変化は tick の幾何スナップ
  ショット比較で検出し全フィルターを更新する。公開値はセッション中スティッキーなので、
  手動設定は存在しない: 仮想スクリーンが一度も描画されていないコールド
  スタート時のみ内部定数の 60° 線形写像を使い、Display Wall 不在時は
  「ウォール=デスクトップ」と見なす。混在 DPI 環境では拡張(DIP)と
  プラグイン(物理 px)の座標差で数度の誤差が出得る（既知の近似）
- センターに合わせるディスプレイ: Display Wall のコンボ `center_display`
  （モニター ID を保存、空 = 自動 = ウォール中央）で、リセンター時に正面へ
  来るディスプレイを選べる。レイアウト公開時にそのモニターの中心 u を
  `nyan_real_wall_center_u()` として公開し、仮想スクリーンのレンダーパスが
  方位角 α に変換（Audio Wall と同じ平面/円筒の式）して pose に
  `quat_from_yaw_y(-α)` を合成 — ワールドを回してスクリーン全体を横方向に
  ずらす（縦は不変）。適用した α は `g_device.screen_yaw_offset_deg` として
  公開され、Audio Wall が方位から差し引いて音と映像の一致を保つ。選択
  ディスプレイがウォールに無い間（除外/未接続）は -1 = 自動にフォールバック
- プロパティ: 角度幅 / 定位モード / 距離減衰（フィルターと共通キー、変更は
  filters_dirty 経由で全子フィルターへ伝播）/ 除外 exe リスト（カンマ区切り）。
  プロパティ画面にキャプチャ中の exe 一覧を情報表示
- 帰結: 空間音響ミックスはモニタリング専用で、配信/録画ミックスには乗らない
  （手動フィルター運用の「モニターのみ」と同じ挙動）
- 制約: 音はアプリ（exe）単位。同一アプリを複数画面に広げても 1 方向に
  まとまる（Windows がウィンドウ/タブ単位の音声分離を提供していないため）。
  ウィンドウを持たないバックグラウンド再生アプリは位置が決められないため
  キャプチャしない

### ブラウザ連携（WebSocket インジェスト）

アプリ単位の限界（ブラウザの全窓が 1 方向にまとまる）を超えるため、
Audio Wall は localhost WebSocket サーバー（`ws_server.cpp`、ポートは
空間音響グループ内の「ブラウザ連携ポート」プロパティ、既定 8796、127.0.0.1
バインド）を持ち、同梱の Chrome 拡張（`tools/chrome-extension/`）から
タブ音声を受ける。サーバーは空間音響グループが ON の間だけ立つ。

- 拡張の content script が `createMediaElementSource()` で媒体要素を
  フックする。これによりタブの音は OS 出力から切り離され、出口が OBS の
  モニタリングだけになる — 「Windows の既定デバイスをメガネ以外へ」の
  回避策すら不要（WS ストリームに関しては）
- プロトコル: 1 接続に複数ストリームを多重化。テキストフレーム =
  `{"type":"meta","stream":id,"label","norm_x","sample_rate","channels","exe"}`
  / `{"type":"close","stream":id}`、バイナリフレーム = u32 LE ストリーム ID +
  16bit インターリーブ PCM。norm_x（仮想デスクトップ上の窓中心、-0.5〜+0.5）は
  拡張側が DIP 座標系で計算して送る（プラグイン側の物理 px と混ぜない）
- 受信側: ストリームごとにプライベートの push 音声ソース
  （`nyan_real_3dof_ws_audio`、CAP_DISABLED でメニュー非表示、
  `obs_source_output_audio` で流し込み）+ 空間フィルターを生成。方位は
  アプリ子と同じ「norm_x × 角度幅」。メタの `exe` はアプリ検出から自動除外
  され、同じブラウザの二重キャプチャを防ぐ
- 寿命: WS close / `close` メッセージ / 5 秒無受信タイムアウトで子を解放。
  サーバーの停止は接続スレッドを join してからウォールを壊す（コールバックは
  children_mutex を取るため、破棄順が安全性の要）
- レイテンシ/安定性: 拡張のキャプチャは AudioWorklet（音声スレッド、
  512 フレーム ≒ 10.7ms バッチ）が主経路で、ページのメインスレッドの
  ジャンクと無縁。ページ CSP で Worklet モジュールを読めない場合のみ
  ScriptProcessor 2048 フレーム（約 43ms、メインスレッド依存）へ
  フォールバック。サーバーは accept 直後に TCP_NODELAY を立て Nagle の
  送信まとめ遅延を排除。受け口ソースは unbuffered（到着即再生）。
  経緯: ScriptProcessor 時代は「キャプチャの穴 × unbuffered」でプチノイズに
  なったため一度バッファありへ退避したが、Worklet 化でキャプチャが
  定常化したため unbuffered を再有効化（負荷の高い環境でポップが出る
  場合はここを戻すのが第一手）
- 拡張はローカルフォールバックを持つ: WS 未接続の間は content script の
  グラフが destination へも出力し（ゲイン 1）、接続が立つとゲイン 0 に
  ランプする。OBS 不在でブラウザが無音になることはない
- 窓位置の追従は 3 系統: `windows.onBoundsChanged` は**移動確定時のみ**
  発火する Chrome 仕様で、レンダラーの `window.screenX` もドラッグ中は
  更新されない（実機確認）。ドラッグ中の追従はサービスワーカーが
  `chrome.windows.get` を 200ms でポーリングして補完する（HWND 矩形は
  移動中も動いており、Chromium はモーダル移動ループ中も UI スレッドの
  タスクを回すため API では生の位置が読める）。content script の
  `window.screenX` 200ms ポーリング（`pos` メッセージ）は他プラットフォーム
  向けの補助として残置。いずれも 8px 以上動いたときだけ送信
- WS サーバーは RFC 6455 のサーバー側最小実装（ハンドシェイクの SHA-1 は
  wincrypt、Base64 は自前、マスク必須・4MB フレーム上限・ping/pong 対応）。
  依存ライブラリなし。スマホリモコン（後述）と共用で、LAN バインド・
  非 Upgrade GET への静的ページ配信・Upgrade ターゲットの検証
  （`on_ws_accept`）・任意スレッドからの `send_text`（コネクション単位の
  送信 mutex）・サーバー側からの個別切断（`close_conn`、ソケット shutdown で
  接続スレッドに通常の teardown をさせる）をオプションで持つ

## スマホリモコン (remote_control.cpp)

メガネをかけたまま PC を操作するためのリモコン。スマホ側はアプリではなく、
プラグインが配信する 1 枚の HTML（`data/remote.html`）をブラウザで開くだけ
（Android / iPhone 両対応）。サーバーは `ws_server` の LAN バインド
（0.0.0.0、ポートは「リモコンのポート」、既定 8797）で、ドックの
「スマホからの操作を受け付ける」（既定 OFF、保存対象）が ON の間だけ立つ。

- 接続フロー: ドックが `remote_control_url()`（`http://<LAN IP>:<port>/?t=<トークン>`）
  を QR コード（ベンダリングした qrcodegen で生成、白パディング付きで
  ダークテーマでも読める）と選択可能な URL テキストで表示する。ページの
  JS は URL クエリのトークンを `ws://host/ws?t=<トークン>` に乗せ、サーバーは
  `on_ws_accept` で照合する（不一致は 403）。ページ自体（静的 UI）は
  トークンなしでも取得できるが、コマンドは一切通らない
- トークンは初回有効化時に CryptGenRandom から base32 12 文字で生成して
  設定に保存。OBS を再起動しても読み取り済み QR が使い続けられる。
  「設定を既定値に戻す」でクリアされ、次回有効化時に再生成（= 旧 QR の失効）
- プロトコル（クライアント → サーバー、JSON テキストフレーム）:
  `{"t":"mv","dx","dy"}` 相対移動 / `{"t":"btn","b":0|1|2,"d":bool}` ボタン /
  `{"t":"whl","n":ノッチ}` OS ホイール / `{"t":"dist","s":ステップ}` 視線ドリー /
  `{"t":"recenter"}` / `{"t":"recal"}`。サーバー → クライアントは
  `{"t":"state","dist":m}`（実効奥行き = スクリーン距離 + 視点前進量）を
  接続時・ドリー/距離/リセンター変更時（ポーリングの差分検出）に送る
- 視線ドリー（`manager_dolly_step`）: 距離ストリップはスクリーン距離では
  なく**視点の並進**を動かす。スクリーン（面）は固定のまま、現在の視線
  レイとスクリーン平面の交点までの残距離を 1 ステップごとに 1.05 倍で
  伸縮し、視点を視線方向へ歩かせる — ズームの不動点が「見ている点」に
  なり、寄るほど正しい透視変形と SBS 視差が付く（マーカー 6DoF 削除時に
  予告した「近づいて大きく見る」の操作系）。蓄積値は
  `viewer_offset_x/y/z`（リセンター後ワールド座標、m）で、レンダラーが
  ワープのレイ原点 `eye_pos_m` に SBS の ±IPD/2 と一緒に加算する。
  残距離の下限は `MIN_DOLLY_GAP_M`（0.15m）、上限は `MAX_DOLLY_GAP_M`
  （20m）。残距離の算出はシェーダーの幾何の正確なミラー — 平面時は
  z=-d の平面交差、湾曲時は同じ円筒（半径 d/curve、(0,0,-d) を通る垂直軸）
  に対する二次方程式で、根の選択（遠い方 `(-b+√)/(2a)` 優先）まで
  シェーダーと一致させる。これによりどの湾曲でも漸近クランプが描画面を
  突き抜けない（平面近似だった初版は強い湾曲の端で突き抜けた）。姿勢が
  無効な間は正面方向への前進にフォールバック。**永続化しない**ランタイム
  状態で、センターに移動 / 再較正 / 機種変更 / 設定リセットでゼロに戻る。
  既知の簡略化: 空間音響の距離減衰と Audio Wall の方位は基準のスクリーン
  距離・原点視点のままで、ドリーには追従しない
- マウス注入は SendInput。移動は相対値を蓄積して GetCursorPos + デルタの
  MOUSEEVENTF_ABSOLUTE で打つ（Windows のポインター加速の影響を受けず、
  タッチパッドの対応付けが決定的になる）。dwExtraInfo の FENCE_WARP_SIG は
  意図的に付けない — リモコンの移動もカーソルフェンスの対象にするため。
  ホイールは小数ノッチを蓄積して WHEEL_DELTA 未満も高分解能イベントで出す
- スクリーン距離は `manager_step_screen_distance`（`SCREEN_DISTANCE_STEP_RATIO`
  = 1.05 倍/ステップの対数スケール、0.1〜10m へクランプ）。将来の PC マウス
  ホイール対応もこの同じ入口を使う想定
- ライフサイクル: `remote_control_sync()` が g_device の設定とサーバー状態を
  突き合わせる（ドックの 500ms ポーリング + 有効化トグル + 設定ロード時の
  `manager_apply_settings`）。bind 失敗（ポート使用中）は 3 秒スロットルで
  リトライ。アンロードは `remote_control_shutdown()`。カーソルフェンスと
  同様、サーバーの起動はドックのポーリング駆動のため Qt なしビルドでは
  設定ロード時の 1 回しか同期されない（実用上はドックありビルド前提）
- ページ側: Pointer Events で 1 本指移動（rAF でフレームごとに合算送信）、
  タップ = 左クリック、静止長押し（350ms・移動 12px 未満）= 左ボタンを
  押したままのドラッグ開始（パッドの枠が緑になり、指を離すと解放。ただし
  250ms 以内の再タッチはドラッグ継続 — パッド端に当たる長いドラッグを
  ストロークで刻める）、2 本指縦 = ホイール、2 本指タップ = 右クリック。
  専用の左右クリックボタン行は持たない（タップ/2 本指タップ/長押し
  ドラッグ+継続で全機能をカバーできるため）。右端の
  ストリップが距離ホイール（上 = 遠ざける、下 = 引き寄せる）。Wake Lock で
  画面消灯を防ぎ、切断時は 1.2 秒間隔で自動再接続。言語は
  navigator.language で日英切替
- スタックボタン対策: サーバーはリモコン由来のボタン押下状態を保持し、
  接続断（ドラッグ中の画面ロック等）・サーバー停止時に押されたままの
  ボタンへ UP を注入する（`release_held_buttons`）
- 接続の生存検知: ページが可視中 4 秒ごとに `{"t":"hb"}` を送り、サーバーは
  12 秒無受信の接続を `close_conn` で追い出す（掃除は `remote_control_sync`
  = ドックのポーリングが駆動）。TCP 任せでは画面ロックや Wi-Fi 断の相手が
  数分単位で「接続中」のまま残るための対策。バックグラウンドのタブは
  タイマーがスロットルされるので、ハートビート停止 = 離席の信号として
  そのまま機能する。ドックは接続が 1 台以上ある間、QR コード/URL の
  代わりに「リモコン接続中 (N 台)」を表示する（QR は接続が消えると再表示。
  トークンは接続で失効しない）
- 単一セッション方針: メガネをかけるのは 1 人なので、認証済みの新しい
  接続が来たら既存セッションを `close_conn` で蹴り出す（newest wins、
  `on_open`）。半死にの旧接続が新しいスキャンを妨げることがない
- トークンの再発行（`remote_control_rotate_token`）: 待機中は QR コードの
  クリック、接続中は「切断して QR を再発行」ボタンが同じローテーション
  操作。新トークンで即サーバーを再起動（リトライスロットルはバイパス）し、
  既存セッションと読み取り済み QR を同時に無効化する。単純な「切断だけ」
  ボタンは意図的に置かない — ページの 1.2 秒自動再接続に負けるため、
  切断はトークン無効化とセットでだけ意味を持つ。実装は「トークンを
  クリアして同期」: 同期が新トークンを生成できれば新 QR、CSPRNG 失敗時は
  フェイルセキュア経路がサーバーを停止する。どちらの結末でも旧トークンは
  確実に失効する（再発行の意図 = 失効を、失敗時も裏切らない）
- セクション折りたたみ時はヘッダーに信号灯サマリーを出す（ステータス
  セクションと同型）: 緑 = 接続中 (N)、黄 = QR 読み取り待機中、赤 =
  サーバー開始失敗。OFF の間は無表示
- PWA の現実: http のため Android のインストール型 PWA（Service Worker /
  installability）は不可（Secure Context 必須。LAN の自己署名 TLS は端末ごとの
  証明書インストールが必要で UX が見合わない）。iOS は
  `apple-mobile-web-app-capable` メタで「ホーム画面に追加」からの
  スタンドアロン（全画面）起動が http でも効くため対応済み。Android は
  ページ右上の全画面ボタン（Fullscreen API）が実用解
- LAN IP の推定はゲートウェイを持つアダプター優先 + プライベート帯
  ボーナスのスコアリング（仮想スイッチの誤選択を避ける）。3 秒キャッシュ。
  見つからない場合はドックに「サーバーを開始できません」を表示
- 設定ミラー（`remote_schema.cpp`）: ページの ⚙ ボタンでドックの
  「ステータス / デバイス操作 / メガネ画面・出力 / 仮想スクリーン /
  詳細設定」を別画面にミラーする。実装は**単一の定義表**で、1 エントリ =
  キー・型（status/toggle/slider/combo/action/text/number）・ドックと同じ
  ロケールキー・範囲・read/write 関数・表示条件（transport_traits）。
  サーバーはここから cfg スナップショット（ローカライズ済みラベル・現在値・
  コンボ選択肢・enabled 込み）を生成し、ポーリングの差分検出（直列化文字列
  比較）で全クライアントへプッシュする。ページは cfg をジェネリックに描画
  するだけで個別項目の知識を持たない（構造シグネチャが変わったときだけ
  DOM 再構築、それ以外は値の in-place 更新。直近 1.5 秒以内に触った行は
  プッシュで上書きしない）。クライアント → サーバーは
  `{"t":"set","k","v"}` で、適用も同じ表経由（ホワイトリスト + 範囲
  クランプ）。書き込みはドックと同じ `g_device` / セッターなので、ドック側
  への反映はドック自身の 500ms ポーリングで自動成立する。Qt が必要な操作
  （メガネへ全画面表示、音声出力先のラッチ再武装）は `projector_request` /
  `monitor_rearm` フラグに積み、ドックのポーリングが消費する
  （brightness_request と同じパターン）。「設定を既定値に戻す」と
  「スマホリモコン」セクションは意図的にミラー対象外（どちらもリモコン
  自身の接続を断ち切れるため）。cfg にはステータス行（HID/メガネ画面/
  IMU/姿勢/サイズ計算結果）も含まれ、スマホが簡易ステータスモニターに
  なる。追加・削除の追従ルールは CONTRIBUTING.md「スマホリモコンの
  設定ミラー」を参照（ページ知識ゼロ + 表の g_device 直接参照により、
  ページ側の追加忘れは構造的に起きず、機能削除はコンパイルエラーで検出）
- セキュリティの前提: トークンを知る同一 LAN 端末 = ユーザー本人の操作と
  みなす。配布物に TLS はなく、盗聴耐性より「QR を読んだ端末しか操作
  できない」ことを保証する設計。初回有効化時は Windows ファイアウォールの
  受信許可（obs64.exe）が必要。多層防御として:
  - フェイルセキュア: `generate_token` は CSPRNG（CryptGenRandom）が失敗
    したら空文字を返し、`remote_control_sync` はトークンが空ならサーバーを
    起動しない（停止する）。推測可能なトークンで口を開けるくらいなら
    動かさない方を選ぶ。`upgrade_target_ok` も空トークンは常に拒否する
  - DNS リバインディング対策: `ws_server` の `on_check_host` で Host
    ヘッダのホスト部が IPv4 リテラルであることを要求する
    （`host_is_ipv4_literal`）。QR の URL は常に LAN IP 直打ちなので正規の
    スマホは通り、攻撃者ドメインを被害者 LAN IP へ rebind して被害者
    ブラウザから叩く経路はドメイン名 Host で弾かれる（GET と Upgrade の
    両方の手前で 403）。Audio Wall 側はループバックバインドのため
    `on_check_host` を設定せず素通し
  - 既知の残存（許容）: トークン比較は非定数時間（LAN のジッタ越しの
    タイミング攻撃は非現実的）、accept 直後・トークン検証前に接続スレッドを
    生成するため LAN 内からの大量 TCP 接続で枯渇し得る（接続数上限なし）。
    いずれも LAN 内の攻撃者を前提とした低リスクとして未対応

## レンダリング挙動

有効なライブ姿勢が無い場合、シェーダーは参照元テクスチャを直接サンプリングするため、
グラスが切断されてもレンダリングがブロックしたりフリーズしたりすることはありません。

仮想スクリーン入力ソースは、選択された参照元をその参照元のベースサイズでオフスクリーンに
描画し、プラグイン自身の出力幅/高さへワープ描画します。レイの FoV は出力サイズのアスペクト比から
計算し、物理スクリーンの横幅は参照元テクスチャのアスペクト比から計算します。そのため、横長の
参照元を選ぶと同じ高さのまま横に広い仮想スクリーンとなり、左右を見渡せる余地が増えます。
